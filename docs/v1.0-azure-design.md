# v1.0 design — Azure deployment

Locks the choices that hurt to change after the first deployment.
Lifecycle, naming, secret handling — fixing these post-deploy means
data migrations and breaking redeployments. Implementation choices
that are easy to refactor (Bicep module boundaries, parameter file
shape) live in the code.

Status: **decisions locked, implementation in progress.** Updates land
inline as the build progresses.

---

## 1. Target architecture (single environment)

## 2. Naming convention

`movate-{env}-{component}` for resources where lowercase + hyphens are
allowed; `movate{env}{component}` (no separators) where the resource
type rejects hyphens (storage accounts, ACR registry names).

| component       | resource type            | name                  |
|-----------------|--------------------------|-----------------------|
| Resource group  | `Microsoft.Resources/...`| `movate-{env}-rg`     |
| Log Analytics   | `Microsoft.OperationalInsights/workspaces` | `movate-{env}-logs` |
| ACR             | `Microsoft.ContainerRegistry/registries`  | `movate{env}acr`      |
| Key Vault       | `Microsoft.KeyVault/vaults`              | `movate-{env}-kv`     |
| Postgres Flex   | `Microsoft.DBforPostgreSQL/flexibleServers`| `movate-{env}-pg`   |
| ACA Env         | `Microsoft.App/managedEnvironments`       | `movate-{env}-cae`    |
| ACA — API       | `Microsoft.App/containerApps`             | `movate-{env}-api`    |
| ACA — Worker    | `Microsoft.App/containerApps`             | `movate-{env}-worker` |

`env` ∈ `{dev, staging, prod}`. Each environment is a separate resource
group; one Bicep deployment per environment. No cross-env shared
resources in v1.0 (no shared registry, no shared key vault) — keeps
each tenancy fully tear-down-able.

## 3. Region

Default: `eastus2`. Parameterizable via the bicep param file. Locked
default because changing later means data migration on Postgres (the
only stateful resource).

## 4. SKU tiers per environment

| resource    | dev / staging       | prod                   |
|-------------|---------------------|------------------------|
| ACR         | Basic               | Standard               |
| Postgres    | Burstable B1ms (1 vCore, 2 GB)   | GeneralPurpose D2ds_v5 (2 vCore, 8 GB) |
| ACA Env     | Consumption only    | Consumption + Dedicated workload profile |
| API replicas| min=1, max=2        | min=2, max=10          |
| Worker reps | min=1, max=2        | min=2, max=20          |
| KV          | Standard            | Standard               |
| Log Analytics | PerGB2018, 30d retention | PerGB2018, 90d retention |

Costs at idle: ~$30-40/mo for dev (Postgres B1ms is the floor); ~$200/mo
for a quiet prod (Postgres D2ds_v5 + idle ACA). Postgres dominates;
ACA Consumption tier scales to zero when idle.

## 5. Secret handling

**Key Vault is the single source of truth.** Container Apps reference
secrets via Key Vault references with managed identity — secrets never
appear in image layers, in deployment outputs, or in environment dumps
of the running container.

Required secrets at deploy time (operator pre-populates):

| name                 | value                                       |
|----------------------|---------------------------------------------|
| `pg-admin-password`  | random 32-char on first deploy; rotated per ops policy |
| `openai-api-key`     | from OpenAI dashboard                       |
| `anthropic-api-key`  | from Anthropic dashboard                    |
| `langfuse-secret-key`| from Langfuse project settings              |
| `langfuse-public-key`| Langfuse public key (technically not secret but co-located with the secret-key for ergonomics) |

Container Apps map these to env vars: `MOVATE_DB_URL` (constructed
from `pg-admin-password` plus deploy-time non-secret pieces),
`OPENAI_API_KEY`, etc.

**API keys for movate's own auth (`mvt_live_...`)** are minted via
`movate auth create-key` AFTER the first deployment and live in the
`api_keys` PG table — they're not Azure secrets. Operators run
`movate auth create-key` against the deployed DB to mint them.

## 6. Networking decision: NO VNet integration in v1.0

VNet integration adds:
- A managed VNet (~$/mo)
- Private endpoints for Postgres + Key Vault + ACR
- Cross-resource DNS resolution
- ~6-8 hours of Bicep complexity

What we get instead in v1.0:
- Postgres Flex with **public IP + firewall rules** restricting access
  to ACA Environment outbound IP (one rule per env)
- Key Vault firewall: same approach
- ACR: public; pull-only via managed identity

Trade-off: traffic leaves the Azure backbone briefly (ACA → Postgres
public IP → backbone). Inside the same region this adds <5ms and zero
$/GB. Defensible for v1.0; revisit in v1.1 when a security review or
compliance requirement forces VNet integration.

## 7. Image pipeline

`Dockerfile` is multi-stage:
- `base`: Python 3.11 + uv + project deps (cacheable layer)
- `runtime`: copies app code, default `CMD` runs `movate serve`
- `worker`: same image, default `CMD` runs `movate worker`

Two ACA apps share the same image; only the `command` differs. Saves
build time and ensures API + worker always run identical code.

`movate deploy` (stage 2) does:
1. `az acr build` (or local docker build + `az acr login` + push)
2. `az containerapp update --image ...` for both API and worker

Stage 1 (this turn) just provisions; the first manual deploy is:
```bash
az acr build --registry movate{env}acr --image movate:0.5.0 .
az containerapp update --name movate-{env}-api --image movate{env}acr.azurecr.io/movate:0.5.0
az containerapp update --name movate-{env}-worker --image movate{env}acr.azurecr.io/movate:0.5.0
```

## 8. Module structure

```
infra/azure/
├── main.bicep                 # orchestrator; targets the resource group
├── main.bicepparam.example    # parameter template (committed; concrete params live elsewhere)
├── modules/
│   ├── loganalytics.bicep     # base for diagnostics
│   ├── acr.bicep              # registry; admin user enabled in dev only
│   ├── keyvault.bicep         # vault + RBAC role assignments to ACA managed identity
│   ├── postgres.bicep         # flex server + admin secret in KV + firewall
│   ├── containerapp-env.bicep # ACA Env wired to Log Analytics
│   ├── containerapp-api.bicep # API container app (ingress: external, port 8000)
│   └── containerapp-worker.bicep  # worker container app (no ingress)
└── README.md                  # operator walkthrough
```

Each module declares `@description`, `@minLength`/`@maxLength` where
applicable, returns IDs / FQDNs the next module needs via `output`.
The orchestrator wires them together.

## 9. Out of scope for v1.0 stage 1

* **VNet integration** — see §6.
* **`movate deploy` CLI** — stage 2.
* **GH-Actions deploy.yml** — stage 2.
* **Auto-scaling rules** — accept ACA defaults; tune in v1.1 once
  there's a real load profile.
* **Backups + DR** — Postgres Flex auto-backups (7d retention) is
  enabled; geo-redundant backup is a v1.1 cost decision.
* **Multi-region failover** — Movate's customers are single-region;
  defer until that changes.
* **Custom domains + TLS** — ACA's `*.azurecontainerapps.io` URL is
  fine for v1.0; custom domain + cert provisioning lands stage 2.
