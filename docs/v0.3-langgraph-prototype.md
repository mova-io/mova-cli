# v0.3 Workflow IR — LangGraph seam findings

**Status:** prototype built, verified, deleted. This doc is what survives.
**Tag of record:** v0.3.0 (commit 1bc481d) for the IR + a follow-up commit
that fixed a runner bug surfaced by the prototype.

## Why this exists

The implementation roadmap calls out as risk #1: *"a bad IR makes Phase 7's
LangGraph swap-in painful."* The mitigation was to build a throwaway IR →
LangGraph compiler in a branch, run it side-by-side with the homegrown
`WorkflowRunner`, and lock in the lessons before declaring v0.3 stable.

The prototype lived briefly in:

- `src/movate/core/workflow/_langgraph_prototype.py`
- `tests/test_langgraph_prototype.py`

Both deleted after this doc was written. Reproduction recipe at the bottom.

## TL;DR

| | |
|---|---|
| **IR shape** | ✅ Maps cleanly. `WorkflowGraph.nodes` → `add_node`, `WorkflowEdge` → `add_edge`, sinks → `END`, `entrypoint` → `set_entry_point`. No structural changes to v0.3's IR are required for v1.1's LangGraph compiler. |
| **State plumbing** | ⚠️ One real semantic mismatch (see Finding 1). v1.1's compiler will need a `merge_dicts` reducer or per-node `{**state, **delta}` wrap. |
| **Failure semantics** | ⚠️ LangGraph raises on node exceptions; our runner stops with partial state. v1.1 must explicitly choose, document, and possibly diverge per node-type (e.g. HITL = pause, agent = raise). |
| **Persistence layer** | ❌ The prototype surfaced a real **double-save bug** in our runner — fixed in this commit. Unrelated to LangGraph but found because the prototype counted records. |
| **Edge / node enums** | ⚠️ `EdgeKind.PARALLEL_FAN_OUT` may not be needed (Finding 4). |

The IR is good. v1.1 is feasible without a schema break. The two non-trivial
items are state plumbing (Finding 1) and HITL via LangGraph checkpointers
(Finding 3).

## Finding 1 — Default state merge is replace, not merge

`StateGraph(dict)` treats the entire dict as a single channel with
**last-write-wins**: a node returning `{"step1": "alpha"}` *replaces* the
whole state with that dict. The homegrown runner's contract is shallow merge.

**Surfaced by:** the first run of the equivalence test failed with
`langgraph_final={'step2': 'beta'}` while `homegrown={'text': 'seed',
'step1': 'alpha', 'step2': 'beta'}`.

**Workaround used in prototype:** node functions return `{**state,
**response.data}`. This is correct but per-node ceremony.

**Recommendation for v1.1:** define a typed `State` with a single
`merge_dicts` reducer:

```python
from typing import Annotated, Any
from typing_extensions import TypedDict

def merge_dicts(left: dict, right: dict) -> dict:
    return {**left, **right}

class WorkflowState(TypedDict, total=False):
    # No statically known keys — workflow state schemas vary. We use a
    # synthetic single-channel Annotated type to install the reducer.
    pass

# Or, more idiomatically, define a custom channel.
```

LangGraph also supports per-key reducers via `Annotated[..., reducer]`, but
since our state schema is JSON Schema (runtime), not Python types, a single
whole-state reducer is the cleanest match.

**Action item:** when v1.1's `LangGraphCompiler` lands, install the reducer
in `compile_to_langgraph` and remove the `{**state, **delta}` ceremony.

## Finding 2 — Double-save bug in `WorkflowRunner` (FIXED)

Counting persisted `RunRecord`s after each prototype run:

- Homegrown runner: 4 records for a 2-node workflow.
- LangGraph prototype: 2 records.

Cause: the runner was calling `executor.execute(...)` (which saves) and
then `_stamp_workflow_link(...)` (which constructed a new `RunRecord` with
a fresh UUID and saved again). Each per-node run got persisted twice.

**Fix shipped in this commit:**

- `Executor.execute(...)` now accepts `workflow_run_id` and `node_id`
  kwargs and stamps them onto the persisted `RunRecord`.
- `WorkflowRunner` threads them through; `_stamp_workflow_link` deleted.
- The runner still synthesizes a `RunRecord` for its `WorkflowResult.runs`
  view (so callers see per-node summaries) but does not persist on the
  success path.
- On the failure path the executor only writes a `FailureRecord`, not a
  `RunRecord`, so the runner explicitly persists an `ERROR`-status row
  for that node — keeps `list_runs(workflow_run_id=...)` complete.

**Lesson:** had we caught this only after v0.3 shipped to a real customer,
their cost reports would have been ~2× inflated for any workflow run.
Prototype found a non-LangGraph bug for free.

## Finding 3 — HITL needs LangGraph checkpointers

LangGraph 1.x supports HITL via `interrupt()` + a checkpointer. There are
SQLite, Postgres, and Redis checkpointers available. Our `NodeType.HUMAN`
maps to a node function that calls `interrupt(...)`, which:

1. Persists state to the checkpointer.
2. Raises a `GraphInterrupt` that the caller must handle.
3. The user resolves the interrupt out-of-band (web form, Slack message).
4. The caller resumes the graph with `Command(resume=user_input)`.

**Implications for v1.1:**

- v1.1 must wire `langgraph.checkpoint.postgres.PostgresSaver` to the
  same Postgres connection movate's storage layer uses (lands v0.5).
- The CLI's "stop with partial state" semantics in v0.3 don't translate
  directly. v1.1 will need a `mdk run --resume <workflow_run_id>` flag.
- `WorkflowRunner` and the LangGraph compiler will diverge on failure:
  the homegrown runner stops on agent failure; LangGraph raises and
  expects you to handle the exception. Document this clearly.

**No action for v0.3.** Capture in the v0.5 Postgres milestone.

## Finding 4 — `EdgeKind.PARALLEL_FAN_OUT` may be redundant

LangGraph supports parallel fan-out as a property of *graph topology*, not
edge kind: just add multiple edges from one node and they execute
concurrently. Our IR's `EdgeKind.PARALLEL_FAN_OUT` would be an edge
property, but the same effect is already implicit when a node has >1
outgoing `SEQUENTIAL` edge.

**Question for v1.1:** does our IR need a distinct `PARALLEL_FAN_OUT` kind
at all, or is it enough that `validate_linear` rejects multiple successors
today and a future `validate_dag` simply allows them?

**Tentative recommendation:** drop `PARALLEL_FAN_OUT` and `PARALLEL_FAN_IN`
from the enum when v1.1 lands. Keep `CONDITIONAL` — it's a real edge-level
property (the `condition` string).

**Rationale:** a smaller enum means fewer ways to be wrong. We'd lose the
ability to *force* sequential semantics on a node-with-multiple-successors
("fire all, wait for all" vs "fire one"), but that's a workflow-level
choice, not an edge-level one.

## Reproduction recipe (if needed in v1.1)

```bash
# 1. Install langgraph as a dev-only dep
uv add --dev langgraph

# 2. Recreate the prototype using this doc + the runner.py code as reference.
#    Key API surface: langgraph.graph.StateGraph(dict), add_node, add_edge,
#    set_entry_point, compile, ainvoke.

# 3. Run the equivalence test:
uv run pytest tests/test_langgraph_prototype.py -v

# 4. Remove langgraph after:
uv remove --dev langgraph
```

The runner's `_project_state` and `_summarize_run` helpers are the only
movate-internal pieces a v1.1 compiler needs to import.

## What v1.1 looks like

Sketch of `src/movate/core/workflow/compilers/langgraph.py`:

```python
from typing import Annotated, Any, TypedDict
from langgraph.graph import StateGraph, END

def merge_dicts(left: dict, right: dict) -> dict:
    return {**left, **right}

class _WorkflowState(TypedDict, total=False):
    pass

class LangGraphCompiler:
    def __init__(self, executor, storage, checkpointer=None): ...

    def compile(self, graph: WorkflowGraph) -> CompiledStateGraph:
        # Install merge_dicts reducer on the state channel.
        # Map each node type to its handler:
        #   AGENT     → _make_agent_node_fn (calls executor.execute)
        #   HUMAN     → _make_human_node_fn (calls interrupt)
        #   TOOL      → _make_tool_node_fn (registry lookup)
        #   FUNCTION  → _make_function_node_fn (callable ref)
        #   SUB_WORKFLOW → recurse via this same compiler
        # Map each edge kind:
        #   SEQUENTIAL  → add_edge
        #   CONDITIONAL → add_conditional_edges
        #   (PARALLEL_* are subsumed by topology, not edge kind)
        ...
```

Phase gate at v1.1: `validate_dag` replaces `validate_linear`; same IR.
