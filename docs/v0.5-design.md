# v0.5 design — server + queue

Locks the schema-shaped decisions before the FastAPI runtime gets built.
These are the choices that hurt to change later (migrations, key formats,
isolation models). Implementation choices that are easy to refactor
(handler organization, error response shape) live in the code reviews.

Status: **decisions locked, implementation in progress.** Updates land
inline as the build happens.

---

## 1. Job queue claim semantics

**Decision:** `SELECT ... FOR UPDATE SKIP LOCKED` (Postgres) /
sqlite-specific `BEGIN IMMEDIATE` + atomic UPDATE (sqlite).

**Rationale:**

- Survives Postgres reboots and worker crashes — the row stays visible
  to the next worker the moment its txn aborts.
- No `LISTEN/NOTIFY` plumbing needed; workers just poll. At v0.5 scale
  (handful of workers, not thousands of jobs/sec) polling is fine.
- Idiomatic, well-understood, doesn't need extension installs.

**Sqlite caveat:** sqlite has no `SKIP LOCKED`. We use a single
short-lived `BEGIN IMMEDIATE` transaction wrapping the
"claim-then-update" pair. Concurrent claimers see "database is locked"
errors and retry — fine for the local-dev use case where there's
typically one worker.

**Schema:** `jobs(job_id, tenant_id, kind, status, agent, input,
result_run_id, error, created_at, claimed_at, completed_at)`. Partial
index `WHERE status = 'queued'` so the claim query is O(queued), not
O(all jobs).

## 2. Multi-tenant isolation

**Decision:** Row-level via mandatory `tenant_id` column on every
record. No schema-per-tenant.

**Rationale:**

- The PRD assumes row-level. Schema-per-tenant scales worse for our
  fleet size and breaks the "one query → multi-tenant analytics"
  ergonomics.
- Existing tables (`runs`, `failures`, `evals`, `workflow_runs`) all
  already carry `tenant_id`. v0.5 just extends this discipline to
  `jobs` and `api_keys`.

**Audit obligation (v1.0 / Phase 6):** every storage method that
returns rows MUST accept a `tenant_id` param and filter by it. The
v1.0 phase has a "tenant isolation audit" task that walks every
storage entry point and confirms the filter is applied. v0.5 is the
first phase where this matters because it's the first phase with a
multi-tenant API surface.

## 3. API key format

**Decision:** `mvt_<env>_<tenant_id_prefix>_<key_id>_<secret>`

| segment | format | purpose |
|---|---|---|
| `mvt` | literal | prefix collision-resistance / grep-ability |
| `env` | `live` \| `test` | hard separation of prod vs CI keys |
| `tenant_id_prefix` | first 8 chars of tenant uuid | human can identify tenant from key |
| `key_id` | 12 chars random base32 | per-key revocation handle |
| `secret` | 32 bytes URL-safe base64 (43 chars) | 256-bit entropy |

Example: `mvt_live_a1b2c3d4_x7k9m2p5n8q1_<43-char-secret>`

**Storage:**

- `api_keys(key_id PRIMARY KEY, tenant_id, env, secret_hash, salt,
  created_at, last_used_at, revoked_at, label)`
- `secret_hash`: `sha256(salt || secret)`. **Not Argon2id** — these
  are non-reusable opaque secrets, not user passwords; bcrypt-class
  hashes are overkill and add per-request latency. The salt prevents
  rainbow tables; the 256-bit entropy makes brute force economically
  infeasible.
- The full key is shown to the user **once** at mint time and never
  again. CLI prints to stderr with a "save this now" warning.

**Verification path:**

1. Parse incoming `Authorization: Bearer <key>` → extract `key_id`.
2. `SELECT secret_hash, salt, revoked_at FROM api_keys WHERE key_id = ?`.
3. Constant-time compare `sha256(salt || presented_secret)` vs
   `secret_hash`.
4. Reject if `revoked_at IS NOT NULL`.
5. Async-update `last_used_at` (UPDATE without await; doesn't block
   the request).

## 4. Workflows in the queue

**Decision:** `POST /run` accepts both agent and workflow payloads.
The job record discriminates via `kind ENUM('agent', 'workflow')`.

**Rationale:**

- Same client UX for both. Caller doesn't care that workflows are a
  composition of agents internally.
- Worker dispatches: `kind = 'agent'` → `Executor.execute(...)`;
  `kind = 'workflow'` → `WorkflowRunner.run(...)`. The runner already
  handles per-node `RunRecord` persistence via the v0.3.1 fix.
- `result_run_id` on the job table holds the produced `run_id` (agent)
  OR `workflow_run_id` (workflow). Type discrimination via `kind`,
  not via column-presence.

**Why not separate endpoints?** Tried in MDK; doubled the handler
count for negligible UX benefit. Single endpoint with payload
discrimination is the same shape as how Anthropic / OpenAI APIs work
for messages vs streaming.

## 5. Job → run linkage

**Decision:** `RunRecord.job_id` (already exists) is the foreign-key
out of the queue. The new `jobs` table has `result_run_id` (nullable)
which mirrors back, so a single `JOIN jobs ON jobs.job_id =
runs.job_id` works either direction.

**Why mirror?** Two reasons:

1. The `runs` table is the source of truth for the executed work; the
   `jobs` table is the queue. Without `result_run_id` you'd need to
   query `runs WHERE job_id = ?` every time you want a job's output —
   wasteful for a polling-heavy `/jobs/{id}` endpoint.
2. Workflows produce many runs per job. `result_run_id` for workflow
   jobs holds the `workflow_run_id`; the per-node runs are still
   reachable via `runs.workflow_run_id`. Same pattern as v0.3.1.

## Implementation order (locked)

1. **Job model + schema + storage methods** ← *this turn*
2. API key model + auth crypto + schema + storage methods
3. FastAPI scaffold (`/healthz`, `/agents`, `/run`, `/jobs/{id}`)
4. Worker claim loop (`runtime/worker.py`)
5. CLI: `mdk serve`, `mdk worker`, `mdk auth ...`
6. PostgresProvider port (last — schema is pinned by then)

Each step has tests that run against `InMemoryStorage` first, then
`SqliteProvider`, then (for #6) `PostgresProvider`. No FastAPI test
that requires a real Postgres until Postgres lands.

## Out of scope for v0.5

Listed so the next person doesn't try to land them in this phase:

- **Streaming responses** (`/run` returning SSE/WebSocket) — v0.6+
  when there's a real consumer.
- **Job priorities / retries** — single FIFO queue. Retries are
  per-run (already in `Executor`), not per-job.
- **Distributed tracing across worker hops** — single-process workers
  for now; the existing `tracer` parents to the same trace.
- **Rate limiting / quotas** — v1.0 paired with the auth audit.
- **WebSockets for live job tailing** — engineers can poll
  `/jobs/{id}` until v1.0+.
- **Postgres LISTEN/NOTIFY** — sqlite parity matters more than
  Postgres optimization.
